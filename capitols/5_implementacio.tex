\chapter{Implementació}
\label{ch:implementacio}

En aquest capítol es detalla el procés de construcció del sistema \textit{Web Core View}. Es descriu l'entorn de treball i s'explora el funcionament tècnic de cada mòdul al mateix temps que s'analitza la seva interfície gràfica d'usuari (UI). Aquest recorregut pràctic permet entendre com els requisits funcionals i l'arquitectura de \textit{software} s'han traduït en una experiència d'usuari intuïtiva.

\section{Entorn de desenvolupament i tecnologies}
Per a la implementació del \textit{frontend}, s'ha configurat un entorn basat en \textbf{Node.js} i l'ecosistema \textbf{Angular CLI 19}. L'estructura del projecte segueix un enfocament de components de tipus \textit{stand\-alone}, eliminant la dependència de mòduls clàssics per optimitzar el procés de \textit{tree-shaking} i la càrrega inicial.

Pel que fa als estils, s'ha utilitzat \textbf{SCSS modular}. S'han definit variables globals per mantenir la consistència visual i s'ha aplicat l'estratègia de detecció de canvis \texttt{OnPush} per garantir que la interfície només es refresqui quan realment hi hagi canvis d'estat significatius.

\subsection{Seguretat (JWT)}
La seguretat es basa en JSON Web Tokens (JWT) \cite{rfc7519}. El flux d'autenticació comença amb una petició POST al servei \textbf{la-core-legacy}. Aquest servei no només valida l'usuari, sinó que interacciona amb \textbf{la-db-engine} per obtenir el perfil de permisos (\textit{grants}) associat. El token JWT resultant conté aquesta metadada xifrada, la qual cosa permet al \textit{frontend} (mitjançant \textit{guards} d'Angular) habilitar o deshabilitar botons de la interfície sense haver de fer consultes addicionals per cada acció de l'usuari.

S'ha implementat un servei (\texttt{auth.service.ts}) que gestiona les crides a l'API. Un cop el gravador valida les credencials, retorna aquest token que s'emmagatzema exclusivament en l'estat en memòria d'un Signal privatiu.

La gestió de la sessió s'ha dissenyat sota un esquema de \textbf{zero-persistence} per maximitzar la seguretat. A diferència de les aplicacions web convencionals que emmagatzemen el JWT a \texttt{localStorage} (vulnerable a atacs \textit{Cross-Site Scripting} o XSS), Web Core View manté el token exclusivament en l'estat d'un \textbf{Signal privat} dins del servei d'autenticació.

Aquesta decisió implica que, si un atacant aconseguís injectar un script maliciós, aquest no tindria una API de navegador on anar a buscar les credencials. El cost d'aquest disseny és que un refresc manual de la pàgina (\texttt{F5}) tanca la sessió, una penalització d'experiència d'usuari (UX) que es considera acceptable i desitjable en una aplicació de seguretat industrial.

  \includegraphics[width=0.85\textwidth]{figures/seq_login.png}
  \caption{Diagrama de seqüència del flux de control d'inici de sessió.}

Per garantir que totes les peticions enviades cap al gravador estan autoritzades (Figura \ref{fig:seq_login}), s'ha definit un interceptor funcional d'Angular 19, anomenat \texttt{authInterceptor}:

\begin{lstlisting}[language=JavaScript, caption={Implementació del HttpInterceptorFn en Angular 19.}]
export const authInterceptor: HttpInterceptorFn = (req, next) => {
  const authService = inject(AuthService);
  const token = authService.getTokenSignal()();
  
  if (token) {
    const authReq = req.clone({
      headers: req.headers.set('Authorization', `Bearer ${token}`)
    });
    return next(authReq);
  }
  return next(req);
};
\end{lstlisting}

Addicionalment, s'han implementat \textbf{Functional Guards} (\texttt{CanActivateFn}) que injecten el servei de seguretat, llegeixen el rol descodificat i bloquegen l'accés a rutes prohibides (per exemple, un operador intentant entrar a configuració). També s'utilitza el \texttt{LanguageInitResolver} per carregar les traduccions abans de renderitzar la vista.

El mecanisme de \textbf{Keep-Alive} actua com un batec de cor (\textit{heartbeat}) coordinat entre el \textit{frontend} i el servei \textbf{la-core-legacy}. S'ha configurat un interval de 60 segons per realitzar peticions síncrones que estenguin la validesa de la sessió al backend. 

Si el servei de backend detecta l'absència de dos pings consecutius, invalida automàticament el token a la base de dades \textbf{la-db-engine}, garantint que les sessions abandonades per talls de connectivitat o tancaments inesperats del navegador no quedin obertes. Al \textit{frontend}, si la petició de Keep-Alive retorna un error 401, l'interceptor funcional (Lògica 5.1) redirigeix l'usuari a la pantalla de \textit{login} de forma immediata, protegint la integritat del sistema.

\begin{lstlisting}[language=JavaScript, caption={Optimització del cicle de vida de \textit{Zone.js} per renovació de \textit{tokens}.}]
private scheduleRenewal(ms: number) {
  this.stopRenewal();
  this.zone.runOutsideAngular(() => {
    this.renewalTimer = setTimeout(() => {
      this.zone.run(() => this.acquireLock());
    }, ms);
  });
}
\end{lstlisting}

\subsection{Detecció de Deltes al Desar (Save)}
Quan l'administrador fa clic al botó de desar (\textit{Save}), en comptes d'enviar tot l'esquema sencer (que saturaria el gravador), s'executa un algorisme recursiu de diferències (\textit{delta payload}) que només extreu i envia els camps modificats:

\begin{lstlisting}[language=JavaScript, caption={Algorisme recursiu de detecció de Deltes per a l'optimització de payloads.}]
export function getObjectDelta(current: any, original: any, path: string = ''): any {
  if (current === original) return undefined;
  if (typeof current !== 'object' || current === null) return current;

  const delta: any = {};
  let hasChanges = false;

  for (const key in current) {
    if (key.startsWith('_')) continue; // Ignora camps visuals
    const currentVal = current[key];
    const originalVal = original ? original[key] : undefined;

    if (JSON.stringify(currentVal) !== JSON.stringify(originalVal)) {
      const diff = getObjectDelta(currentVal, originalVal, `${path}.${key}`);
      if (diff !== undefined) {
        delta[key] = diff;
        hasChanges = true;
      }
    }
  }
  return hasChanges ? delta : undefined;
}
\end{lstlisting}

\begin{lstlisting}[language=JavaScript, caption={Orquestració reactiva de protocols de vídeo mitjançant Angular Effects.}]
constructor() {
  effect(() => {
    const streamName = this.cameraStreamName();
    const signalAvailable = this.hasSignal();

    this.webrtcService.closeStream();
    this.hlsService.closeStream();

    if (!signalAvailable) return;

    if (getStreamingMethod() === 'hls') {
      this.hlsService.startStream(this.videoPlayer().nativeElement, streamName);
    } else {
      this.webrtcService.startStream(streamName);
    }
  });
}
\end{lstlisting}

\section{Sistema d'Alarmes i Comunicació asíncrona}
Per a la gestió de les notificacions, s'ha evitat l'obertura de múltiples connexions per cada component que necessiti dades. S'ha implementat un \textbf{servei de missatgeria centralitzat} que manté un únic canal WSS obert amb el mòdul \textit{Core Notifier}. 

Aquest servei utilitza un flux de dades basat en el patró \textit{Observable Shared}, on els diferents components de la interfície (el mosaic de vídeo, la taula d'alarmes i la barra d'estat) se subscriuen a la mateixa connexió. Aquesta arquitectura redueix dràsticament la càrrega de treball del Proxy Invers \textbf{NGINX} i el consum de sockets al sistema operatiu del gravador, permetent una escalabilitat del sistema sense degradació del rendiment.
Per alertar d'intrusions o problemes detectats pels relés, el sistema implementa un canal de \textbf{WebSockets (WSS)} basat en un patró observador asíncron.

\begin{figure}[htbp]
  \centering
  \includegraphics[width=0.95\textwidth]{figures/ui_alarms.png}
  \caption{Taula de monitoratge d'alarmes en temps real i botons d'assentiment.}

\end{figure}

Cada cop que l'aplicació rep un missatge pel \textit{socket}, la taula d'esdeveniments (Figura \ref{fig:ui_alarms}) s'actualitza de manera reactiva, afegint la nova línia. Les alarmes passen per un cicle de vida (Figura \ref{fig:state_alarm}), podent ser assentides (\textit{acknowledged}) mitjançant el botó \textit{Ack} al final de cada fila, indicant al sistema global que l'operador ja està tractant la incidència.

\begin{figure}[!htbp]
  \centering
  \includegraphics[width=0.7\textwidth]{figures/state_alarm.png}

Per donar robustesa davant talls de xarxa, la connexió del WebSocket utilitza \textit{RxJS} amb un \textit{backoff exponencial}:

\begin{lstlisting}[language=JavaScript, caption={Connexió WebSocket resilient amb backoff exponencial en RxJS.}]
this.socket$.pipe(
  retry({
    delay: (error, retryCount) => {
      if (!this.userService.isAuthenticated()) {
        return throwError(() => new Error('Connexió tancada per pèrdua de credencials.'));
      }
      return timer(RECONNECT_INTERVAL_MS * retryCount); // Backoff intel·ligent
    },
  })
).subscribe({
  next: (message) => this.messagesSubject$.next(message)
});
\end{lstlisting}

El flux de dades asíncron segueix un camí crític des del maquinari fins a la interfície:
\begin{enumerate}
    \item El servei \textbf{la-core-alarm} detecta un canvi d'estat en els relés o una intrusió i publica un esdeveniment al bus D-Bus intern.
    \item El mòdul de \textit{Protocol Bridging} (Core Notifier) captura aquest missatge, el transforma a format JSON i l'injecta al túnel de \textbf{WebSockets} gestionat per NGINX.
    \item Al \textit{frontend}, el servei \texttt{WebSocketService} rep el paquet i actualitza un \texttt{Signal} d'Angular. 
    \item Gràcies a la reactivitat granular de la versió 19, la fila corresponent de la taula d'alarmes es renderitza de nou instantàniament sense afectar la reproducció de vídeo activa, garantint que l'operador rebi la alerta en un temps mitjà de 200ms.
\end{enumerate}

Garantint l'operativa completament desconnectada (\textit{air-gapped}) de la xarxa exterior de moltes d'aquestes màquines de seguretat, tota la documentació oficial d'ús i guies ràpides està introduïda en la mateixa memòria web del dispositiu (Figura \ref{fig:ui_documentation}). L'usuari només ha de clicar a la fila corresponent i l'arxiu PDF es descarregarà al seu PC sense requerir accés a portals web corporatius externs.
