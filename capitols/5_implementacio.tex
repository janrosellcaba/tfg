\chapter{Implementació}
\label{ch:implementacio}

En aquest capítol es detalla el procés de construcció del sistema \textit{Web Core View}. Es descriu l'entorn de treball i la implementació de cada mòdul funcional, destacant la lògica reactiva i la integració amb el gravador.

\section{Entorn de desenvolupament i tecnologies}
Per a la implementació del \textit{frontend}, s'ha configurat un entorn basat en \textbf{Node.js} i l'ecosistema \textbf{Angular CLI 19}. L'estructura del projecte segueix un enfocament de components de tipus \textit{stand\-alone}, eliminant la dependència de mòduls clàssics per optimitzar el procés de \textit{tree-shaking} i la càrrega inicial.

Pel que fa als estils, s'ha utilitzat \textbf{SCSS modular}. S'han definit variables globals per mantenir la consistència visual i s'ha aplicat l'estratègia de detecció de canvis \texttt{OnPush} per garantir que la interfície només es refresqui quan realment hi hagi canvis d'estat significatius.

\section{Mòdul d'Autenticació i Core}
La seguretat de l'aplicació es gestiona de manera centralitzada en la capa \textit{Core} mitjançant la gestió de sessions amb JSON Web Tokens (JWT) \cite{rfc7519}. S'ha implementat un servei d'autenticació (\texttt{auth.service.ts}) que gestiona les crides a l'API de \textit{login}. Un cop el gravador valida les credencials del client, retorna un token signat digitalment. Aquest token s'emmagatzema en l'estat en memòria d'un Signal privatiu de l'aplicació per evitar atacs d'escriptura maliciosa de tipus \textit{Cross-Site Scripting} (XSS) associats a l'ús de \textit{localStorage}.

Per tal de garantir que totes les peticions enviades cap al gravador estan correctament autoritzades, s'ha definit un interceptor funcional d'Angular 19, anomenat \texttt{authInterceptor} (\texttt{HttpInterceptorFn}). Aquest interceptor s'encarrega d'examinar la petició, extreure el token actiu i, en cas de ser-hi present, clonar la petició per injectar la capçalera HTTP de seguretat corresponent:

\begin{lstlisting}[language=JavaScript, caption={Implementació del HttpInterceptorFn en Angular 19.}]
export const authInterceptor: HttpInterceptorFn = (req, next) => {
  const authService = inject(AuthService);
  const token = authService.getTokenSignal()();
  
  if (token) {
    const authReq = req.clone({
      headers: req.headers.set('Authorization', `Bearer ${token}`)
    });
    return next(authReq);
  }
  return next(req);
};
\end{lstlisting}

\subsection{Protecció de rutes i Functional Guards}
Per tal d'assegurar que l'accés a determinades seccions està degudament restringit segons el perfil de l'usuari (Operador o Administrador), s'han implementat **Functional Guards** basats en la nova especificació d'Angular 19, que simplifica el disseny evitant la declaració de classes clàssiques.

La protecció de rutes de configuració es realitza mitjançant un guàrdia funcional de tipus \texttt{CanActivateFn}. Aquest guàrdia injecta el servei de seguretat, llegeix el rol descodificat en el token JWT del client i bloqueja l'accés al router web si l'usuari té un rol insuficient (per exemple, un operador intentant accedir al menú de configuració del sistema). En cas de bloqueig, el guard redirigeix l'usuari a la pàgina de visor de vídeo i emet un missatge de notificació emergent.

A més, s'ha implementat el resolver \texttt{LanguageInitResolver} per carregar els fitxers de traducció abans de renderitzar la interfície principal i així evitar parpellejos en la internacionalització.

\section{Implementació del Motor Schema-Driven}
Aquesta és la part més complexa de la implementació. El motor ha de ser capaç de transformar un esquema JSON abstracte en una interfície totalment funcional.

\subsection{El component genèric de configuració}
S'ha creat el component \texttt{Config\-Input\-Component}, el qual actua de forma recursiva segons l'estructura del JSON rebut. La decisió de quin tipus d'element de formulari s'ha d'instanciar per a cada paràmetre es resol dinàmicament en temps d'execució analitzant la propietat \texttt{type} i les metadades de restricció de l'esquema JSON mitjançant un flux de selecció múltiple (\textit{switch-case}):

\begin{description}
  \item[\texttt{type: "boolean"}] $\rightarrow$ Es mapeja cap al component \texttt{ToggleComponent} (un botó de commutació gràfic actiu/inactiu).
  \item[\texttt{type: "integer" o "float"}] $\rightarrow$ Si l'esquema inclou metadades de rang limitat (\texttt{min} i \texttt{max}), es renderitza el component \texttt{SliderComponent} per a un control tàctil precís. Si no es defineix un rang tancat, s'instancia el component \texttt{NumericInputComponent} amb filtrat actiu de caràcters de teclat.
  \item[\texttt{type: "string"}] $\rightarrow$ Si el camp posseeix una propietat \texttt{enum} de valors tancats, es renderitza el component \texttt{SelectComponent} (un menú desplegable). Si conté una propietat de validació com \texttt{pattern} (expressió regular), es mapeja cap al component \texttt{TextInputComponent} amb validació activa de regex en temps real (per a formats d'adreça IP, DNS, etc.).
  \item[\texttt{type: "object"}] $\rightarrow$ S'aplica la recursivitat cridant el component contenidor \texttt{ConfigGroupComponent}, que agrupa lògicament les subpropietats i genera una pestanya o secció col·lapsable a la interfície.
  \item[\texttt{type: "array"}] $\rightarrow$ S'instancia el component \texttt{ListConfigComponent}, que gestiona de manera dinàmica l'escriptura, inserció i eliminació de conjunts de dades indexades (com llistes d'outputs de relés o presets de càmeres).
\end{description}

\subsection{Optimització del payload mitjançant deltes de canvis}
A causa de les limitacions de maquinari del videogravador embedded, enviar l'esquema sencer de configuració de càmeres o usuaris en cada petició de desat (\textit{save}) suposaria una sobrecàrrega inacceptable en el processament XML/JSON del backend i una transferència innecessària d'ample de banda. Per solucionar-ho, s'ha dissenyat un algorisme recursiu de detecció de diferències (\textit{deep delta detection}) que compara l'estat actual editat per l'usuari amb l'estat original obtingut de l'NVR. L'algorisme genera un objecte de canvis mínims (\textit{delta payload}) que només s'envia si s'han detectat modificacions reals, permetent a més ignorar els camps interns (que comencen amb guió baix \texttt{\_}) utilitzats només per a la renderització visual del frontend.

\begin{lstlisting}[language=JavaScript, caption={Algorisme recursiu de detecció de diferències (Delta) per a l'optimització de payloads.}]
export function getObjectDelta(
  current: any,
  original: any,
  forceFullKeys: string[] = [],
  path: string = ''
): any {
  if (current === original) return undefined;

  if (typeof current !== 'object' || current === null) {
    return current;
  }

  const delta: any = {};
  let hasChanges = false;

  for (const key in current) {
    // S'ignoren camps interns de gestió visual de l'esquema
    if (key.startsWith('_')) continue;

    const currentVal = current[key];
    const originalVal = original ? original[key] : undefined;
    const currentPath = path ? `${path}.${key}` : key;

    // Comprovació de si la propietat requereix transmissió completa forçada
    const isProtected = forceFullKeys.some((pattern) => {
      if (pattern === key || pattern === currentPath) return true;
      const regexStr = pattern.replace(/\./g, '\\.').replace(/\{[^}]*\}/g, '[^.]+');
      const regex = new RegExp(`(^|\\.)${regexStr}$`);
      return regex.test(currentPath);
    });

    if (JSON.stringify(currentVal) !== JSON.stringify(originalVal)) {
      if (isProtected) {
        delta[key] = currentVal;
        hasChanges = true;
      } else {
        const diff = getObjectDelta(currentVal, originalVal, forceFullKeys, currentPath);
        if (diff !== undefined) {
          delta[key] = diff;
          hasChanges = true;
        }
      }
    }
  }

  return hasChanges ? delta : undefined;
}
\end{lstlisting}

\subsection{Optimització de Cicle de Vida i Fil d'Execució}
El servei de bloqueig de configuració (\texttt{ConfigLockService}) requereix un mecanisme de batec de cor (\textit{heartbeat}) que renovi periòdicament el token de bloqueig al backend abans que expiri. Executar temporitzadors ordinaris de JavaScript (\texttt{setInterval}) dins de la zona d'Angular (Zone.js) provocaria que el framework disparés el cicle de detecció de canvis global a tota l'aplicació en cada renovació, degradant dràsticament el rendiment del processament de vídeo en directe. Per evitar-ho, s'ha dissenyat un algorisme de renovació asíncrona que s'executa fora de la zona d'Angular mitjançant \texttt{NgZone.runOutsideAngular}, reentrant a la zona de detecció de canvis de forma explícita només quan es rep la confirmació d'èxit de renovació.

\begin{lstlisting}[language=JavaScript, caption={Optimització del cicle de vida de \textit{Zone.js} per renovació de \textit{tokens}.}]
import { Injectable, NgZone, inject } from '@angular/core';

@Injectable({ providedIn: 'root' })
export class ConfigLockService {
  private zone = inject(NgZone);
  private renewalTimer: any;

  // Planifica la renovació de forma eficient fora de la detecció de canvis global
  private scheduleRenewal(ms: number) {
    this.stopRenewal();
    
    // S'executa fora d'Angular per evitar disparar Zone.js innecessàriament
    this.zone.runOutsideAngular(() => {
      this.renewalTimer = setTimeout(() => {
        // Només reentrem a la zona quan cal actualitzar l'estat reactiu de l'aplicació
        this.zone.run(() => this.acquireLock());
      }, ms);
    });
  }

  public stopRenewal() {
    if (this.renewalTimer) {
      clearTimeout(this.renewalTimer);
      this.renewalTimer = null;
    }
  }
}
\end{lstlisting}

\subsection{Orquestració Reactiva amb Signals i Effects}
La visualització de vídeo requereix una sincronització perfecta entre l'estat de la càmera (com ara la pèrdua de senyal), el canal de streaming seleccionat (codec, resolució) i l'element HTML \texttt{<video>} del DOM. En lloc d'utilitzar mètodes tradicionals basats en temporitzadors (\textit{polling}) o subscripcions complexes que poden provocar fugues de memòria, s'ha utilitzat el mecanisme reactiu de les \textbf{Angular Signals} combinat amb un \texttt{effect}. Aquest efecte d'orquestració s'executa automàticament de forma asíncrona cada vegada que canvia algun dels valors reactius dels quals depèn (\texttt{cameraSignal}, \texttt{selectedCodecId}, \texttt{hasSignal}), gestionant de forma neta l'obertura i tancament d'instàncies dels serveis de WebRTC o HLS.

\begin{lstlisting}[language=JavaScript, caption={Orquestració reactiva de vídeo mitjançant Angular Effects.}]
// Constructor de l'orquestrador de vídeo basat en Angular Signals
constructor() {
  effect(() => {
    const streamName = this.cameraStreamName();
    const signalAvailable = this.hasSignal();
    const player = this.videoPlayer();

    // S'alliberen i es tanquen nítidament les connexions i sockets anteriors
    this.webrtcService.closeStream();
    this.hlsService.closeStream();

    if (!signalAvailable) {
      console.warn('Stream bloquejat: Pèrdua de senyal detectada (NO SIGNAL).');
      return;
    }
    if (!streamName) {
      console.warn('Stream bloquejat: Nom de flux invàlid per a la càmera.');
      return;
    }
    if (!player) {
      console.warn('Stream bloquejat: Element de reproducció de vídeo no renderitzat.');
      return;
    }

    // Inicialització dinàmica del protocol seleccionat segons configuració de l'aplicació
    if (getStreamingMethod() === 'hls') {
      this.hlsService.startStream(player.nativeElement, streamName);
    } else {
      this.webrtcService.startStream(streamName);
    }
  });
}
\end{lstlisting}

\subsection{Estratègia de reconnexió amb Backoff exponencial}
La gestió d'esdeveniments i alarmes de videovigilància és un servei de caràcter crític. Per garantir que l'operador rebi les notificacions en temps real fins i tot després d'una caiguda de xarxa transitòria o un reinici del videogravador, s'ha programat una estratègia de connexió altament resilient utilitzant la reactivitat de RxJS. El servei de connexió encapsula el cicle de vida del WebSocket de manera que, en cas de desconnexió o pèrdua de pols, utilitza l'operador \texttt{retry} amb una estratègia de temporització intel·ligent (\textit{backoff exponencial}) per evitar saturar la CPU del videogravador i s'assegura que no es reintenti la connexió si l'usuari s'ha desautenticat de manera voluntària.

\begin{lstlisting}[language=JavaScript, caption={Connexió WebSocket resilient amb backoff exponencial en RxJS.}]
this.socket$ = webSocket<ServerSentMessage>(config);
this.socket$
  .pipe(
    tap({
      error: (error) => console.error('Error crític detectat al canal WebSocket:', error),
    }),
    retry({
      delay: (error, retryCount) => {
        // Atura immediatament els intents si l'usuari ha tancat la sessió
        if (!this.userService.isAuthenticated()) {
          return throwError(() => new Error('Connexió tancada per pèrdua de credencials.'));
        }

        // Si el gravador està reiniciant-se de forma controlada, es difereix l'intent
        if (this.maintenanceService.isSystemRebooting()) {
          console.log(`Reintent de connexió deferred per reinici. Intent: #${retryCount}`);
          return timer(RECONNECT_INTERVAL_MS);
        }

        console.warn(`Pèrdua de connectivitat. Reintentant connexió en ${RECONNECT_INTERVAL_MS}ms (Intent #${retryCount})`);
        return timer(RECONNECT_INTERVAL_MS);
      },
    }),
    catchError((error) => {
      console.error('WebSocket tancat definitivament després dels intents màxims.');
      this.connectionStatus$.next(false);
      this.detailedConnectionState$.next('DISCONNECTED');
      return EMPTY;
    })
  )
  .subscribe({
    next: (message: ServerSentMessage) => {
      this.messagesSubject$.next(message);
    }
  });
\end{lstlisting}
