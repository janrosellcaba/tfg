% ---  DIAPOSITIVA 9 — Pàgina de Directe  ---
\begin{frame}{Pàgina de Directe: Vídeo en Temps Real}
  \begin{columns}[T]
    \begin{column}{0.54\textwidth}
      \begin{center}
        \includegraphics[width=\textwidth,keepaspectratio]{../figures/interficies/cameras.png}
      \end{center}
    \end{column}

    \begin{column}{0.43\textwidth}
      \textbf{Sota el capó:}
      \begin{itemize}
        \item \textbf{WHEP} (RFC 9716): POST \texttt{/stream/<cam>/whep} → SRTP/UDP + DTLS + ICE.
        \item Latència: \textbf{150--300 ms}. ActiveX+RTSP: 800--2000 ms.\\
              \small Els rangs \textbf{no se solapen}.
        \item Fallback \textbf{HLS automàtic} en $\leq$ 5 s si UDP bloquejat. Transparent per l'operador.
        \item \textbf{Angular Signals}: frame càmera 3 → actualitza \textit{un} component. \textbf{35\% menys CPU} (i3-8100T, Windows 10 IoT).
      \end{itemize}
    \end{column}
  \end{columns}

  \note{
    Quan l'operador obre la pàgina de directe, Angular fa un POST a l'endpoint WHEP de cada càmera. WHEP, el WebRTC HTTP Egress Protocol, estandaritza la negociació amb una sola crida HTTP. A partir d'aquí s'estableix la connexió UDP directa amb xifrat DTLS i ICE per a la travessia de NATs.

    El resultat: 150 a 300 mil·lisegons de latència. L'antiga plataforma ActiveX basada en RTSP oscil·lava entre 800 ms i 2 segments. Els rangs no se solapen: el pitjor cas de WebRTC és millor que el millor cas d'ActiveX.

    Si UDP és bloquejat per un firewall corporatiu, Angular detecta el timeout en 5 segments i commuta automàticament a HLS. L'operador mai veu un error.

    I gràcies als Angular Signals, quan arriba un frame de la càmera 3, s'actualitza únicament el component d'aquella càmera. Zone.js hauria rellegit tot el DOM. Resultat mesurat: 35\% menys CPU en un terminal Intel Core i3-8100T sota Windows 10 IoT, amb 4 càmeres en continu.
  }
\end{frame}

% ---  DIAPOSITIVA 10 — Pàgina de Gravacions  ---
\begin{frame}{Pàgina de Gravacions: Timeline i H.265}
  \begin{columns}[T]
    \begin{column}{0.54\textwidth}
      \begin{center}
        \includegraphics[width=\textwidth,keepaspectratio]{../figures/interficies/recordings.png}
      \end{center}
    \end{column}

    \begin{column}{0.43\textwidth}
      \textbf{Sota el capó:}
      \begin{itemize}
        \item Gravacions en \textbf{FS2}: blocs bruts sense contenidor. Cap reproductor web els llegeix directament.
        \item \textbf{Trans-muxing} a \texttt{la-video-engine}: llegeix FS2 → recupera \textbf{SPS/PPS/VPS} → empaqueta \textbf{MPEG-TS} → HLS/TCP.
        \item \textit{Trans-muxing} $\neq$ \textit{transcoding}: canviem l'embolcall, no el bitstream. \textbf{CPU quasi zero}.
        \item \textbf{H.265}: \textbf{50\% menys disc} que H.264. Crític en gravadors 24/7.
      \end{itemize}
    \end{column}
  \end{columns}

  \note{
    L'operador selecciona un segment temporal i reprodueix. El que veu és fluid; el que passa al darrere és complex.

    Les gravacions al disc estan en format FS2 propietari: blocs bruts sense contenidor, sense capçaleres estàndard. Cap reproductor web pot llegir-los directament.

    El microservei la-video-engine fa trans-muxing en temps real: llegeix el bloc FS2, recupera les capçaleres SPS, PPS i VPS que contenen la informació de descodificació ---que de vegades no estan en cada segment sinó que cal recuperar del keyframe anterior---, i les injecta davant del bitstream. Finalment, empaqueta tot en format MPEG-TS per a HLS.

    La clau és que no estem recomprimint el vídeo, només canviem l'embolcall. La CPU del gravador no s'ha d'esforçar en operacions de codificació. H.265 ocupa un 50\% menys de disc que H.264 per la mateixa qualitat: factor crític en gravadors que funcionen les 24 hores.
  }
\end{frame}

% ---  DIAPOSITIVA 11 — Pàgina d'Alarmes  ---
\begin{frame}{Pàgina d'Alarmes: Push i Persistència}
  \begin{columns}[T]
    \begin{column}{0.54\textwidth}
      \begin{center}
        \includegraphics[width=\textwidth,keepaspectratio]{../figures/interficies/alarms.png}
      \end{center}
    \end{column}

    \begin{column}{0.43\textwidth}
      \textbf{Sota el capó:}
      \begin{itemize}
        \item \texttt{la-core-alarm}: push \textbf{WSS} persistent. Sensor salta → icona actualitzada en $<$ \textbf{1 s}. Signals: s'actualitza \textbf{una} icona.
        \item Reconnexió amb \textbf{backoff exponencial} (1s, 2s, 4s, 8s...): protegeix la CPU d'allaus de reconnexions.
        \item \textbf{SQLite VFS} (\texttt{remote\_vfs}): escriptures via Unix Socket al \texttt{la-db-engine}, sectors \textbf{512 bytes} atòmics.
        \item Mode \textbf{WAL} + \texttt{synchronous=NORMAL}: garanties \textbf{ACID}. Zero corrupció en talls elèctrics.
      \end{itemize}
    \end{column}
  \end{columns}

  \note{
    El sistema d'alarmes requeria resoldre dos problemes independents: la notificació en temps real i la persistència davant de talls de corrent.

    Per a la notificació: connexió WebSocket segura persistent. Quan un sensor salta, la-core-alarm ho emet immediatament per WebSocket a tots els operadors connectats. Gràcies als Signals, s'actualitza únicament la icona de l'alarma afectada. Si la connexió cau, la reconnexió automàtica amb backoff exponencial protegeix la CPU del gravador d'allaus de reconnexions simultànies.

    Per a la persistència: el driver VFS personalitzat remote\_vfs redirigeix totes les operacions d'SQLite a través d'un Unix Domain Socket cap al la-db-engine, l'únic procés amb permisos d'escriptura al disc. Les escriptures s'alineen a sectors de 512 bytes per garantir atomicitat davant de talls de corrent sobtats. El mode WAL ofereix garanties ACID completes, verificades en simulacions de tall elèctric forçat.
  }
\end{frame}

% ---  DIAPOSITIVA 12 — Pàgina de Configuració  ---
\begin{frame}{Pàgina de Configuració: Schema-Driven i Lock}
  \begin{columns}[T]
    \begin{column}{0.52\textwidth}
      \begin{center}
        \includegraphics[width=\textwidth,keepaspectratio]{../figures/interficies/configuration.png}
      \end{center}
    \end{column}

    \begin{column}{0.44\textwidth}
      \begin{block}{Motor Schema-Driven}
        \small Backend envia metadades: \texttt{\_type}, \texttt{\_allowedValues}, \texttt{\_min}, \texttt{\_max}. Angular \textbf{auto-genera} el formulari.\\[0.2em]
        \texttt{ipaddr} $\to$ input regex IP $\cdot$ \texttt{list} $\to$ \texttt{<select>} $\cdot$ \texttt{integer} $\to$ rang [min,max]\\[0.2em]
        Nou firmware $\Rightarrow$ \textbf{zero canvis al frontend}.
      \end{block}

      \vspace{0.3em}

      \begin{alertblock}{Config Lock Pessimista}
        \small POST \texttt{/lock} $\to$ token \textbf{32-char hex}.\\
        Expiració automàtica \textbf{60--120 s}.\\
        Altres usuaris: \textbf{HTTP 423 Locked}.\\
        Si el tècnic tanca el portàtil: el lock caduca sol.
      \end{alertblock}
    \end{column}
  \end{columns}

  \note{
    El formulari que l'usuari veu és 100\% generat automàticament. Cap enginyer de frontend ha escrit cap camp de configuració.

    El motor Schema-Driven funciona així: el backend no envia només dades, envia metadades. Cada camp porta el tipus, els valors permesos, els límits. Angular llegeix l'esquema i renderitza el formulari complet. Un camp de tipus ipaddr genera un input amb regex de validació IP. Un camp de tipus list genera un select amb exactament les opcions permeses. Un integer genera un input numèric amb rang ja configurat. El motor té 10 tipus de camp que cobreixen el 100\% de la configuració.

    L'impacte empresarial és directe: un nou model de NVR amb 20 paràmetres nous → zero canvis al frontend. Zero PR, zero deploy.

    El Config Lock pessimista resol el problema de concurrència que vam plantejar al disseny. Quan un tècnic entra a configurar, fa POST a /lock i obté un token de 32 caràcters hexadecimals. Tota escriptura posterior ha d'incloure'l. Si tanca el portàtil, el lock expira en 60-120 segments sol. Altres usuaris reben HTTP 423 i resten en mode lectura automàticament.
  }
\end{frame}
