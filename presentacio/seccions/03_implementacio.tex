    Si la xarxa corporativa bloqueja UDP ---que és el transport natural de WebRTC---, el sistema commuta automàticament a HLS, que va per TCP sobre el port 443 de NGINX, sempre disponible. L'operador no veu cap error.

    A la dreta de la vista de càmeres hi ha un panel lateral que mostra l'estat de les entrades i sortides físiques del gravador ---sensors i relés--- en temps real.

    I gràcies als Angular Signals, quan canvia l'estat d'una càmera, s'actualitza únicament aquella cel·la del mosaic. Resultat mesurat: 35\% menys CPU en un Intel i3-8100T amb 4 càmeres en continu.
  }

% ---  DIAPOSITIVA — Vídeo en Directe 2/2  ---
\begin{frame}{WebRTC — Connexió}
  \begin{columns}[T]
    \begin{column}{0.38\textwidth}
      \vspace{1.5em}
      \begin{itemize}
        \item \textbf{1 POST WHEP} $\to$ negociació completa
        \item \textbf{SRTP/UDP}: xiframent extrem a extrem
        \item UDP bloquejat $\to$ \textbf{HLS} automàtic
        \item Patró \textbf{Strategy}: canvi transparent
      \end{itemize}
    \end{column}

    \begin{column}{0.59\textwidth}
      \begin{center}
        \vspace{-2cm}
        \includegraphics[width=\textwidth, height=1\textheight, keepaspectratio]{../figures/diagrames/activity_video_connection.png}
      \end{center}
    \end{column}
  \end{columns}

  \note{
    El protocol WHEP és la peça clau que fa possible WebRTC sense cap configuració al client. Tota la negociació es fa amb una sola crida HTTP POST estàndard.

    Angular envia un SDP offer al backend: descriu les capacitats del navegador ---codec H.264, transport UDP/TLS. MediaMTX respon amb 201 Created i un SDP answer amb els candidats ICE: les adreces IP i ports UDP disponibles per establir la connexió. A partir d'aquí, el navegador i el servidor fan el handshake DTLS per intercanviar les claus de xifratge, i a partir d'aquest moment tot el vídeo va per SRTP sobre UDP, xifrat extrem a extrem.

    El diagrama de la dreta és la lògica de reconnexió. Angular monitoritza contínuament l'estat ICE de la connexió. Si la connexió cau, comença un cicle de reconnexió automàtic. Si detecta que UDP és bloquejat ---el timeout és de 5 segons---, commuta al camí de fallback HLS, que és TCP sobre el port 443 de NGINX, sempre disponible.

    El patró Strategy és el que permet tenir fallback transparent. Tant WebRTCService com HlsService implementen la mateixa interfície StreamingStrategy. El component de la graella no sap ni li importa quin dels dos està usant: crida startStream i espera events de connectionState.

    Quan Angular detecta el timeout de 5 segons per UDP, destrueix el WebRTCService i n'injecta un HlsService. El component no canvia ni una línia de codi. L'operador mai veu un error, simplement el vídeo continua. Això és el desacoblament real.

    L'impacte de Signals: el connectionState és un Signal. Quan la càmera 3 canvia d'estat ---connecting, connected, error---, Angular actualitza únicament el component de la cel·la 3 del mosaic. Zone.js hauria disparat un cicle de detecció de canvis global i rellegit tot el DOM. En un gravador amb CPU compartida entre vídeo, disc i web, aquesta diferència és real: 35\% menys CPU mesurat en un Intel Core i3-8100T amb 4 càmeres en continu sota Windows 10 IoT.

    El panel lateral d'I/O: quan l'operador obre el panel, Angular envia un missatge de subscripció al WebSocket de la-core-notifier. Qualsevol canvi d'estat elèctric al hardware ---un sensor que s'activa, un relé que commuta--- arriba al frontend en mil·lisegons via D-Bus $\to$ la-core-notifier $\to$ WebSocket $\to$ Signal $\to$ element IoItem.
  }
\end{frame}

  \begin{columns}[T]

      \begin{center}
        \includegraphics[width=\textwidth, height=0.74\textheight, keepaspectratio]{../figures/interficies/recordings.png}

    \end{column}

    \begin{column}{0.38\textwidth}
      \vspace{0.8em}
      {\small\textcolor{gray}{El repte: \textbf{FS2} propietari al disc}}

      \vspace{0.5em}
      \begin{itemize}
        \item Trans-muxing: \textbf{FS2 $\to$ MPEG-TS}
        \item $\neq$ transcoding: \textbf{CPU $\approx$ 0}
        \item \textbf{H.265}: 50\% menys disc
        \item Línia de temps interactiva
      \end{itemize}
    \end{column}
  \end{columns}

  \note{
    La pàgina de gravacions permet a l'operador seleccionar un rang temporal i reproduir el clip al navegador. La línia de temps és interactiva: es pot avançar, retrocedir i descarregar el segment.

    El repte és que les gravacions al disc estan en format FS2, un format propietari de Lanaccess: blocs de bitstream bruts sense capçaleres estàndard. Cap MediaSource Extensions, cap hls.js, cap reproductor web pot llegir-los directament.

    La solució és trans-muxing en temps real: la-video-engine converteix FS2 a MPEG-TS sobre la marxa, sense mai recomprimir el vídeo. El bitstream H.264 o H.265 passa intacte: l'únic cost és l'empaquetament. Gràcies a H.265, les gravacions de 24 hores ocupen un 50\% menys de disc que amb H.264: factor crític en gravadors que funcionen contínuament.
  }
\end{frame}

% ---  DIAPOSITIVA — Gravacions 2/2  ---
\begin{frame}{Gravacions — Trans-muxing i Durabilitat}
  \begin{columns}[T]
    \begin{column}{0.48\textwidth}
      \textbf{Per cada segment demanat:}
      \begin{enumerate}
        \item Llegir bloc \textbf{FS2} del disc
        \item Recuperar \textbf{SPS/PPS/VPS} si cal
        \item Empaquetar com \textbf{MPEG-TS}
        \item Servir via \textbf{HLS/TCP}
      \end{enumerate}
    \end{column}

    \begin{column}{0.48\textwidth}
      \textbf{H.265 + Durabilitat:}
      \begin{itemize}
        \item 1 TB: 10 dies $\to$ \textbf{20 dies}
        \item Mode \textbf{WAL}: lectura i escriptura simultànies
        \item Tall elèctric $\to$ bloc anterior \textbf{íntegre}
      \end{itemize}
    \end{column}
  \end{columns}

  \note{
    Expliquem el com del trans-muxing pas a pas.

    Quan l'operador selecciona un rang temporal, Angular fa GET del manifest M3U8 passant el token de sessió, el timestamp d'inici i de fi. La-video-engine localitza els blocs FS2 que cobreixen el rang mitjançant la biblioteca d'indexació interna i genera el fitxer de manifest dinàmicament: cada entrada correspon a un interval de vídeo real. Si detecta un tall de més de 3 segons entre dos buffers ---per exemple, una gravació per events interrompuda--- injecta la directiva EXT-X-DISCONTINUITY, que informa el reproductor del navegador que la línia temporal s'ha trencat: sense això, el descodificador quedaria congelat.

    Per cada segment, la la-video-engine carrega el bloc FS2 en un buffer temporal de 8 MB. Trans-muxing significa empaquetar el bitstream en un contenidor MPEG-TS sense tocar ni un sol bit del vídeo: no hi ha decodificació, no hi ha recompressió. En un dispositiu embedded amb CPU limitada, això és crític.

    La complexitat de l'H.265 és que les capçaleres SPS, PPS i VPS ---que el descodificador necessita per inicialitzar-se--- no es repeteixen a cada frame: estan una sola vegada al principi del clip. Si l'operador fa un seek enmig d'un clip, el backend retrocedeix a l'índex del primer buffer del fitxer per recuperar les capçaleres de referència i les injecta davant del segment. Sense aquest mecanisme, el descodificador del navegador rebria un flux incomplet i no podria reproduir res.

    Sobre la durabilitat: un gravador NVR és un dispositiu que pot perdre la llum en qualsevol moment. FS2 escriu els fotogrames en blocs contigus de mida fixa directament sobre la partició raw: si l'alimentació es perd a meitat d'una escriptura, la integritat del bloc anterior queda preservada per disseny.

    Per a la base de dades d'índex de gravacions, la solució és el driver VFS personalitzat remote\_vfs. SQLite té una capa d'abstracció de sistema de fitxers: totes les operacions d'I/O passen per funcions xRead, xWrite. El remote\_vfs reimplementa aquesta interfície i redirigeix cada operació via Unix Domain Socket cap a la-video-engine, que és l'únic procés del sistema amb permisos d'escriptura al disc físic. Cap altre procés pot comprometre la integritat. El mode WAL permet que els operadors reprodueixin gravacions mentre el sistema continua gravant, sense cap bloqueig.

    Sobre H.265: mateixa qualitat visual, la meitat de bits. En termes pràctics: un gravador amb 1 TB passa de 10 dies de retenció a 20 dies sense ampliar el disc. Per a un client d'infraestructura crítica que necessita conservar 30 dies, la diferència és comprar un disc addicional o no.
  }
\end{frame}

\begin{frame}{Alarmes}
  \begin{columns}[T]
    \begin{column}{0.55\textwidth}
      \begin{center}
        \includegraphics[width=\textwidth, height=0.74\textheight, keepaspectratio]{../figures/interficies/alarms.png}
      \end{center}
    \end{column}

    \begin{column}{0.40\textwidth}
      \vspace{0.8em}
      \begin{itemize}
        \item Sensor salta $\to$ push \textbf{WSS} $<$ 1\,s
        \item Backoff: 1\,s, 2\,s, 4\,s\ldots màx \textbf{30\,s}
        \item Persistència: \textbf{SQLite + WAL}
        \item \textbf{Cap alarma perduda}
      \end{itemize}
    \end{column}
  \end{columns}

  \note{
    La pàgina d'alarmes és la vista de reactivitat en temps real del sistema. Quan salta un sensor ---moviment, intrusió, pèrdua de senyal de càmera---, la notificació ha d'arribar a tots els operadors connectats en mil·lisegons.

    L'arquitectura és push pura: el microservei la-core-alarm manté una connexió WebSocket segura permanent amb cada client Angular. Quan el sensor dispara, la-core-alarm emet l'event immediatament per WebSocket a tots els operadors. Cap polling. Cap latència de cicle.

    Si la connexió cau ---per exemple, un reinici del gravador---, la reconnexió usa backoff exponencial: 1s, 2s, 4s, fins a un màxim de 30s. Sense backoff, en un entorn amb 50 operadors, tots intentarien reconnectar-se alhora i colapsarien la CPU del gravador just quan encara està arrencant serveis.

    Gràcies als Signals, quan arriba un event d'alarma, s'actualitza únicament la icona de l'alarma afectada. El component no sap de les altres càmeres: total isolament de re-render.
  }
\end{frame}

% ---  DIAPOSITIVA — Alarmes 2/2  ---
\begin{frame}{Alarmes — Pipeline}
  \begin{columns}[T]
    \begin{column}{0.46\textwidth}
      \small
      \textbf{3 fils desacoblats:}
      \begin{enumerate}
        \item \textbf{D-Bus} $\to$ cua \textit{thread-safe}
        \item gRPC $\to$ \textbf{persistència} SQLite
        \item WSS $\to$ \textbf{tots} els operadors
      \end{enumerate}
      \vspace{0.5em}
      \begin{itemize}
        \item gRPC lent $\to$ \textbf{WSS no s'atura}
      \end{itemize}
    \end{column}

    \begin{column}{0.50\textwidth}
      \begin{center}
        \vspace{-0.4cm}
        \includegraphics[width=\textwidth, height=0.84\textheight, keepaspectratio]{../figures/diagrames/alarms_data_flow.png}
      \end{center}
    \end{column}
  \end{columns}

  \note{
    El cor del sistema d'alarmes és el microservei la-core-alarm, escrit en C++ multithreading. El disseny clau és un pipeline de tres fils independents que es comuniquen exclusivament a través de cues thread-safe.

    El fil de D-Bus manté una connexió bloquejant al bus del sistema escoltant passivament qualsevol senyal ALARM\_NOTIFY de la interfície com.lanaccess.sysevents. Quan el maquinari dispara una alarma ---un sensor de moviment, una entrada digital, una pèrdua de senyal---, aquest fil construeix una estructura AlarmData i la diposita a la cua global\_alarm\_queue sense processar-la.

    El fil de persistència es desperta immediatament, agafa l'element de la cua i el desa a la base de dades via gRPC cap a la-db-engine. Quan la persistència confirma èxit, l'event passa a la global\_emission\_queue.

    El fil d'emissió agafa l'event de la cua i el difon en JSON a tots els clients WebSocket connectats en aquell moment.

    El benefici del desacoblament és clau: si la base de dades va lenta o gRPC es bloqueja temporalment, el fil de D-Bus continua rebent senyals del maquinari i no es perd cap event. Si el WebSocket d'un client és lent, no bloqueja la persistència. Cada fil és independent.

    Angular rep els events per WebSocket, actualitza un Signal per a la fila afectada, i re-renderitza únicament aquell component. La resta de la UI no es toca.

    Sobre la resiliència: el backoff exponencial evita que 50 operadors reconnectin simultàniament: 1s, 2s, 4s fins a 30s. La poda automàtica: un SQL Trigger s'executa després de cada inserció i elimina els registres més antics si se superen 10.000 entrades. Zero codi especial al backend.

    Quant a l'Ack: quan l'operador confirma una alarma, el backend no tan sols la marca com a llegida per a ell. Envia ALARM\_READ\_SUCCESS a tots els operadors connectats. Si n'hi ha cinc davant de cinc pantalles, tots cinc veuen la fila actualitzada en temps real. Consistència multi-operador garantida.
  }
\end{frame}

\begin{frame}{Sistema de Configuració}
  \begin{columns}[T]
    \begin{column}{0.58\textwidth}
      \begin{center}
        \includegraphics[width=\textwidth, height=0.74\textheight, keepaspectratio]{../figures/interficies/config1.png}
      \end{center}
    \end{column}

    \begin{column}{0.38\textwidth}
      \vspace{0.3em}
      {\small\textcolor{gray}{El problema: nou firmware $\to$ canvis al frontend}}

      \vspace{0.5em}
      \textbf{Motor Schema-Driven:}
      \begin{itemize}
        \item Backend envia \textbf{metadades}, no dades
        \item 10 tipus: \texttt{ipaddr}, \texttt{list}, \texttt{boolean}\ldots
        \item Formulari \textbf{100\% auto-generat}
        \item Nou NVR $\to$ \textbf{zero canvis} al frontend
      \end{itemize}
    \end{column}
  \end{columns}

  \note{
    La pantalla que veieu és la configuració d'una càmera IP. L'operador veu un formulari ben estructurat amb checkboxes, desplegables i camps de text. El que és sorprenent és com s'ha construït: Angular no té cap formulari definit al codi. No hi ha cap plantilla HTML que descrigui quins camps han d'existir.

    Quan l'operador obre la configuració, Angular fa un GET al backend i rep un arbre JSON ple de metadades. Cada camp porta un \_type. El component ConfigInputComponent llegeix aquest \_type i renderitza l'element HTML correcte automàticament.

    Hi ha 10 tipus que cobreixen el 100\% de la configuració. L'impacte directe: un NVR amb 20 paràmetres nous en el pròxim firmware és zero canvis al frontend. Zero PR. Zero deploy. El gravador és l'única font de veritat.

    Exemple real: el camp \texttt{disabled} és boolean, renderitza un checkbox. El camp \texttt{class} és list amb \_allowedValues VIVOTEK/ONVIF/RTSP, renderitza un select amb exactament aquelles opcions. El camp \texttt{address} és ipaddr, un input amb validació regex d'IP. El camp \texttt{port} és numeric amb \_minValue 1 i \_maxValue 65535, que Angular trasllada als atributs HTML5 min i max. I \texttt{password} és un input de tipus password amb restriccions de longitud i política de seguretat.

    Cap d'aquesta lògica és codi Angular específic per a càmeres. És declaració pura: el schema diu QUÈ és cada camp, el ConfigInputComponent sap COM renderitzar-ho. Deu tipus en total, cobrint el 100\% del sistema.
  }
\end{frame}

% ---  DIAPOSITIVA — Config Lock Pessimista 2/2  ---
\begin{frame}{Config Lock Pessimista}
  \begin{columns}[T]
    \begin{column}{0.46\textwidth}
      {\small\textcolor{gray}{El problema: dos admins editen alhora $\to$ dades corruptes}}

      \vspace{0.5em}
      \textbf{Pessimistic Lock:}
      \begin{itemize}
        \item \textbf{POST /lock} $\to$ token 32-char
        \item Interceptor \textbf{transparent} a tot PUT
        \item Segon admin: \textbf{HTTP 423} $\to$ mode lectura
        \item Tanca portàtil $\to$ expira \textbf{sol} (60–120\,s)
      \end{itemize}
    \end{column}

    \begin{column}{0.50\textwidth}
      \begin{center}
        \vspace{-2cm}
        \includegraphics[width=\textwidth, height=1\textheight, keepaspectratio]{../figures/diagrames/seq_config_lock.png}
      \end{center}
    \end{column}
  \end{columns}

  \note{
    La concurrència és un problema real: un tècnic de camp i un administrador poden obrir la configuració simultàniament i sobreescriure els canvis de l'altre. El Pessimistic Lock ho elimina completament.

    En entrar a configuració, Angular fa automàticament POST a /lock. El servidor retorna un token de 32 caràcters i un temps de vida: 60 o 120 segments depenent del node. A partir d'aquí, tot PUT a /config.* porta el token: no ho fa el component ni el servei, sinó un interceptor HTTP global transparent. Cap component sap de la seva existència.

    La renovació és cada expiresIn/2 fora de Zone.js per no disparar cicles de detecció de canvis innecessaris. Si un segon operador intenta adquirir el lock: HTTP 423 Locked, i el frontend el posa automàticament en mode lectura via un Signal. Si el tècnic tanca l'ordinador, el lock expira sol i es lliura sense cap intervenció manual.
  }
\end{frame}
